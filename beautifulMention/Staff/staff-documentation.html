1. Login

/staff/login

Figure 1 — Staff panel login page with email and password fields

Description

The Login screen serves as the authentication entry point for the Staff panel, used by both Supervisors and Reviewers. Built on the Filament 5 authentication system, the page presents an emerald-themed login form requiring an email address and password. All form submissions include Laravel CSRF token protection to prevent cross-site request forgery attacks.

Multi-Factor Authentication (MFA): The Staff panel enforces MFA. After a successful email/password validation, users who have a configured app_authentication_secret are prompted to enter a one-time code from their authenticator application before being granted access to the dashboard.

Functional Flow

  1. User navigates to /staff/login.
  2. User enters email address and password.
  3. System validates credentials against the users table.
  4. If valid and the user has an app_authentication_secret, the MFA one-time code prompt is displayed.
  5. Upon successful authentication, the user is redirected to the Staff Dashboard (/staff).
  6. A session record is created in the sessions table.

2. Password Reset

/staff/password-reset

Figure 2 — Password reset request form

Description

The Password Reset screen allows staff users who have forgotten their credentials to request a password reset link. The form collects the user's email address and, if valid, dispatches a tokenized reset link to that address. The reset token is stored in the password_reset_tokens table and is time-limited to prevent abuse.

Functional Flow

  1. User clicks the "Forgot your password?" link on the login page.
  2. User is navigated to /staff/password-reset.
  3. User enters their registered email address.
  4. System checks whether the email exists in the users table.
  5. If valid, a tokenized reset link is generated and sent to the provided email.
  6. A record is created in the password_reset_tokens table with the email, hashed token, and timestamp.
  7. User follows the link from email to set a new password.

3. Dashboard

/staff

Figure 3 — Staff dashboard with evaluation stats, supervised projects, and review assignments widgets

Description

The Dashboard is the main hub for staff users after authentication. It provides a comprehensive overview of the user's current workload through a set of role-aware widgets. The dashboard content adapts based on the authenticated user's roles (Supervisor, Reviewer, or both).

Header Widgets

PendingEvaluationsWidget: Displays three stat cards showing the count of evaluations assigned to the current user, segmented by status:
  • Pending — Evaluations not yet started
  • Draft — Evaluations in progress (saved but not submitted)
  • Submitted — Evaluations completed and submitted

Footer Widgets

The footer area contains two role-specific widgets:

  • SupervisedProjectsWidget (visible only to users with the Supervisor role): A summary table listing the user's supervised projects with columns for project title, semester, student count, evaluation progress, and project status.
  • ReviewAssignmentsWidget (visible only to users with the Reviewer role): A summary table listing the user's review evaluations with columns for project name, semester, rubric, and evaluation status.

Database Verification

status varchar(255) NOT NULL Stat card segmentation (pending/draft/submitted)
projects supervisor_id bigint FK NOT NULL, FK users.id Filters supervised projects to current user
status varchar(255) NOT NULL Project status badge in supervised projects widget
users id bigint PK AUTO INCREMENT Authenticated user identity

5. My Review Assignments

/staff/my-review-assignments

Figure 5 — Table of review evaluations assigned to the current user with fill order and status

Description

The My Review Assignments page displays a data table of all evaluations assigned to the currently authenticated user in the Reviewer role. This page is only accessible by users with the Reviewer role; non-reviewer users are denied access.

Table Columns

Column Description Features
Project Title Title of the associated project Searchable
Semester The academic semester of the project —
Rubric Name The rubric template used for this evaluation —
Fill Order The sequence number derived from phase rubric rules Determines dependency order
Status Current evaluation status Pending Draft Submitted

Fill Order Enforcement

Dependency Rule: The fill order mechanism ensures that evaluations are completed in the correct sequence. Specifically, supervisors must submit their evaluations before reviewers can begin theirs. The fill_order value is derived from the phase_rubric_rules table. A reviewer's Fill Assessment action is only enabled when all evaluations with a lower fill_order for the same project have been submitted.

Actions

  • Fill Assessment: Opens the Evaluation Form. Available only when the evaluation status is Pending or Draft and the fill order dependency is satisfied (all preceding evaluations are submitted).
  • View Assessment: Opens the Evaluation Form in read-only mode. Available only when the evaluation status is Submitted.
  • Project Detail: Links to the Project Detail page for the associated project.

7. Evaluation Form

/staff/evaluation/{evaluation}

Figure 7 — Dynamic evaluation form with group and individual criteria sections, score levels, and feedback fields

Description

The Evaluation Form is a dynamic assessment page built from rubric criteria associated with the evaluation's rubric template. The page title displays the rubric name, and the subheading shows the project title and evaluator role. The form is constructed using the BuildsEvaluationForm trait, which dynamically generates form fields based on the rubric's criteria and their score levels.

Form Structure

Two-Section Layout: The form is divided into two distinct sections based on the is_individual flag on each criterion:
  • Group Criteria (is_individual = false): Scored once for the entire project group. Each criterion appears as a single fieldset.
  • Individual Criteria (is_individual = true): Scored per student. Each criterion is nested by criterion first, then by student fieldsets within it.

Per-Criterion Fields

Field Type Description
Score Level Dropdown (Select) Options from score_levels table showing label + score_value. Selecting a level auto-populates the Score Awarded field.
Score Awarded Numeric Input Range: 0 to max_score (from criterion). Step: 0.01. Displayed with a /max suffix indicator. Can be manually adjusted after auto-population.
Feedback Textarea Free-text feedback for the specific criterion.

General Feedback

A general feedback textarea appears at the bottom of the form, allowing the evaluator to provide overall comments about the project or student performance beyond individual criteria.

Score Level Auto-Population

When a score level is selected from the dropdown, the score_awarded field is automatically populated with the corresponding score_value from the score_levels table. The evaluator may then manually adjust this value if needed, as long as it remains within the valid range (0 to max_score).

Read-Only Mode

When the evaluation status is Submitted, all form fields are disabled (read-only). The evaluator can view their scores and feedback but cannot modify them. The Save Draft and Submit action buttons are hidden in this state.

Status Transitions

Auto-Upgrade on First Open: When an evaluation with a Pending status is first opened in the form, its status is automatically upgraded to Draft. This indicates that the evaluator has begun reviewing the evaluation.
Transition Trigger Validation
Pending → Draft First time the form is opened None (automatic)
Draft → Draft Save Draft button clicked None (partial data allowed)
Draft → Submitted Submit button clicked All criteria must have scores assigned

Actions

  • Save Draft: Saves the current form data without submitting. Allows partial completion; not all criteria need scores. Status remains Draft.
  • Submit: Validates that all criteria have scores assigned, then marks the evaluation as Submitted. Once submitted, the evaluation is locked and becomes read-only.

Authorization and Fill Order

  • Authorization: Only the assigned evaluator (evaluator_id matching the current user) can access the evaluation form.
  • Fill Order Check: If the fill order dependency is not satisfied (i.e., evaluations with a lower fill_order for the same project have not yet been submitted), the system returns a 403 Forbidden response, preventing premature access.
